# PDF 파서 동작 분석 및 리스크 포인트

이 문서는 현재 프로젝트의 “건설기술인 경력증명서 PDF → JSON” 파싱 흐름이 어떻게 동작하는지와, 코드 기준으로 확인된 **미비점/오류 발생 가능 지점**을 정리한 것입니다.

---

## 전체 파싱 흐름(엔드투엔드)

메인 오케스트레이션은 `main.py`의 `parse_full_document()`가 담당합니다.

- **입력**
  - 기본 입력은 **PDF 파일 경로**이며, 내부적으로 `pdfplumber.open()`으로 페이지를 엽니다.
  - `--from-json/--pick-json` 모드는 **PDF 파싱 없이** JSON을 읽어 `--excel/--validate` 등 후처리만 수행합니다.
  - 입력 파일이 ZIP 등 **PDF가 아닌 경우는 지원하지 않으며**, `main.py`에서 명시적으로 에러 처리합니다.

- **1) 제1쪽 계열(정확히는 ‘기술경력’ 섹션 전까지 통합)**
  - 엔트리: `parsers/page_1_parser.py: parse_page_1()`
  - 대상: 인적사항/서류출력일자/등급/국가기술자격/학력/교육훈련/상훈/벌점및제재사항/근무처
  - 방식 요약
    - 최대 8페이지까지 텍스트를 누적하되 `"1. 기술경력"`을 만나면 중단하여 상단 섹션을 통합 파싱합니다.
    - **텍스트 기반 파싱**으로 뼈대를 먼저 만들고,
    - 이후 `parsers/section_parsers.py`의 **표 기반 파서(`extract_tables`)** 결과로 상훈/자격/근무처/벌점·제재 등을 보강/정규화합니다.
    - 중복 제거(키 조합), “해당없음” 처리, 회사명 `現:`/흡수합병 표기 정규화 등 보정 로직이 포함돼 있습니다.

- **2) 기술경력 구간(제2쪽 파서, 여러 페이지)**
  - 엔트리: `parsers/page_2_parser.py: parse_page_2()`를 `tech_start..tech_end` 범위 페이지별 호출 → 리스트 누적
  - 페이지 경계 보정: `parsers/page_2_parser.py: merge_cross_page_tech_overviews()`
  - 핵심 설계
    - `참여기간_시작일 ~ 참여기간_종료일` 및 `(인정일수)(참여일수)` 블록(`date_blocks`)을 **1순위로 “절대 누락 없이”** 확보
    - 사업명/발주자/공사종류/직무분야/담당업무/직위/개요/공법 등은
      - 텍스트 주변 문맥(로컬 컨텍스트) + 표 데이터(사업명 매칭)로 보강
    - 문서 레이아웃이 특이한 “홍성달형(흐름형)”은 별도 흐름 파서로 파싱한 뒤 품질 점수로 채택 여부를 결정합니다.
    - `┖→`(개요 연장) 페이지 시작은 중복 파싱을 막기 위해 상단 블록을 스킵하고, 이후 `merge_cross_page_tech_overviews()`가 이전 페이지의 마지막 경력 개요에 이어붙입니다.

- **3) 건설사업관리 및 감리경력 구간(제3쪽 파서, 여러 페이지)**
  - 엔트리: `parsers/page_3_parser.py: parse_page_3()`를 `cm_start..cm_end` 범위 페이지별 호출 → 리스트 누적
  - 방식 요약
    - 표 기반으로 프로젝트 기본 필드를 만들고
    - 텍스트 기반으로 날짜/일수 블록을 추출해 병합합니다.
    - 마지막 CM 페이지 하단의 “최근 1년 통계”를 `_parse_recent_1y_service_stats()`로 추출해 모든 CM 행에 반영합니다.

- **4) 요약 페이지(분야별 참여기간 인정일)**
  - 엔트리: `parsers/page_summary_parser.py: parse_page_summary()`
  - `summary_start`부터 문서 끝까지 텍스트를 합쳐 `_parse_summary_text()`로
    - `공사종류별인정일수`
    - `직무전문분야별인정일수`
    를 추출합니다.

- **5) 검증 및 오류 기록**
  - `main.py: count_pdf_items()`로 PDF 원본 텍스트 기반 기대 건수를 추정하고,
  - 파싱 결과와 비교해 불일치 시 `result["_파싱오류"]`에 기록합니다.
  - 기술경력은 `_verify_tech_career_per_page_against_pdf()`로 **페이지별 “유효 ~ 개수 vs 파싱 건수”** 비교 결과도 기록합니다.
  - 또한 `log_technical_career_field_issues()`로 기술경력/CM 섹션의 `사업명` 타입/공란 등을 검사해 `_파싱오류`에 추가합니다.

## 미비한 부분 / 오류 발생 가능 지점(우선순위)

### 1) 입력 정책: PDF만 지원(비-PDF 입력은 명시적 에러)

- **현상**
  - 파싱 파이프라인은 `pdfplumber` 기반이며, 입력은 **PDF(.pdf)** 로 고정합니다.
  - `main.py`의 `_parse_one_pdf()`가 확장자 및 ZIP 여부를 검사해 **비-PDF 입력을 즉시 에러로 종료**합니다.
- **영향**
  - 입력 포맷 혼선이 제거되어, 운영/배치에서 “지원되는 줄 알고 넣었다가 실패”하는 케이스를 예방합니다.

### 2) 요약 페이지 파서가 `summary_start`부터 “끝까지 합쳐서” 파싱 → 섹션 혼입 시 오탐 위험

- **현상**
  - `parse_page_summary()`는 `page_num..EOF`를 모두 합칩니다.
  - `_parse_summary_text()`는 특정 헤더 이후 라인을 훑는 방식이라, 문서 변형으로 유사 패턴이 뒤 페이지에 등장하면 섞일 수 있습니다.
- **영향**
  - 요약이 마지막이 아닌 문서(추가 부록/설명 페이지가 붙는 케이스)에서 **과대 추출/오탐** 가능성이 있습니다.
- **개선 방향**
  - 요약 섹션 종료 앵커를 더 강하게 잡고(다음 대섹션 제목 등),
  - 합치는 페이지 수에도 상한을 두는 편이 안전합니다.

### 3) 검증용 “원본 PDF 기대 건수” 카운팅이 일부 섹션에서 정규식 기반 → 거짓 불일치 가능

- **현상**
  - `count_pdf_items()`는 섹션별로 서로 다른 기준을 사용합니다(일부는 파서 결과 기반, 일부는 정규식 기반).
  - 특히 근무처/자격 등은 텍스트 줄바꿈/레이아웃 영향을 크게 받습니다.
- **영향**
  - 실제 파싱은 맞는데 검증 카운팅이 틀려 `_파싱오류`가 과다 기록될 수 있습니다(운영 노이즈 증가).
- **개선 방향**
  - 가능한 한 검증도 “해당 섹션 파서와 동일한 규칙/추출 경로”로 기대 건수를 산출하거나,
  - 검증 결과의 신뢰도를 구분(참고용/강제 실패용)하는 체계가 필요합니다.

### 4) `┖→`(개요 연장) 제거 로직의 안전장치 부족(희박하지만 “정상 사업명 절단” 리스크)

- **현상**
  - `merge_cross_page_tech_overviews()`는 다음 페이지 첫 행 `사업명`이 cont 텍스트로 시작하면 해당 prefix를 제거합니다.
- **영향**
  - cont 텍스트가 짧거나 흔한 문구일 때, 극단적으로는 정상 사업명이 잘릴 여지가 있습니다(확률은 낮지만 리스크는 존재).
- **개선 방향**
  - cont 길이 하한(너무 짧으면 제거 보류),
  - cont가 규격/수치/목록형 텍스트일 때만 제거 등 가드 조건을 추가하면 더 안전합니다.

### 5) `extract_text()` vs `extract_words()` 혼용 시 “라인 오프셋 기반 스킵”이 어긋날 수 있음

- **현상**
  - 기술경력에서 상단 스킵(`parse_start_offset`)은 `extract_text()` 기반 줄 목록에서 계산합니다.
  - 하지만 `date_blocks`는 가능한 경우 `extract_words()` 기반 라인 스트림을 사용합니다.
- **영향**
  - 두 라인 스트림의 “줄 구조”가 다르면 같은 오프셋이 다른 의미가 되어 스킵이 과/부족될 수 있습니다.
- **개선 방향**
  - 스킵 판정/적용을 동일 라인 스트림에서 수행하거나,
  - 오프셋이 비정상(범위 초과/구조 불일치)일 때 0으로 되돌리는 방어가 유효합니다.

### 6) 성능/안정성: `pdfplumber.open()` 반복 호출(배치에서 병목 + 실패 지점 증가)

- **현상**
  - `parse_full_document()` 내부에서도 여러 번 PDF를 열고,
  - `parse_page_2/3`도 호출마다 PDF를 다시 엽니다.
- **영향**
  - 배치 파싱 성능 저하, 리소스/IO 오류 확률 증가.
- **개선 방향**
  - 상위에서 PDF 객체(또는 페이지 리스트)를 열어두고 하위 파서에 전달하는 구조로 변경하면 개선됩니다.

### 7) 미구현/미사용 코드 존재

- 미구현이거나 현재 사용되지 않는 코드는, 유지보수 혼선을 줄이기 위해 제거/격리하고 문서에서도 현황을 최신화하는 편이 안전합니다.

3. **검증(카운팅) 로직의 거짓 경보로 인한 운영 노이즈**
